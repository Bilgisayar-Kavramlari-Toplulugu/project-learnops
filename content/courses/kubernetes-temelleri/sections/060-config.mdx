# ConfigMap ve Secret

Uygulamalar farklı ortamlarda (development, staging, production) farklı konfigürasyonlarla çalışır. Veritabanı adresi, API URL'leri, şifreler... Bunları image içine gömmek hatalıdır. Kubernetes, bu sorunu **ConfigMap** ve **Secret** ile çözer.

## 12-Factor App: Konfigürasyonu Koddan Ayır

Modern uygulama geliştirme prensiplerinden biri olan **12-Factor App**, konfigürasyonun ortam değişkenleri aracılığıyla uygulamaya iletilmesini önerir:

```
❌ Yanlış:
Dockerfile:
  ENV DB_HOST=prod-db.internal
  ENV DB_PASSWORD=supersecret123

→ Her ortam için farklı image build etmek gerekir
→ Şifreler image'a gömülür (Git geçmişinde kalır!)

✅ Doğru:
Image ortamdan bağımsız
Konfigürasyon runtime'da inject edilir (ConfigMap/Secret)
```

## ConfigMap

Hassas olmayan yapılandırma verilerini saklar. Düz metin veri, konfigürasyon dosyaları, URL'ler, özellik bayrakları...

### ConfigMap Oluşturma Yöntemleri

**Yöntem 1: YAML ile**

```yaml
# app-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  # Anahtar-değer çiftleri
  APP_ENV: "production"
  LOG_LEVEL: "info"
  MAX_CONNECTIONS: "100"
  API_BASE_URL: "https://api.example.com"

  # Çok satırlı değer (dosya içeriği gibi)
  app.properties: |
    spring.datasource.url=jdbc:postgresql://db:5432/myapp
    spring.jpa.show-sql=false
    server.port=8080

  nginx.conf: |
    server {
        listen 80;
        location / {
            proxy_pass http://localhost:8080;
        }
    }
```

**Yöntem 2: Komut satırından**

```bash
# Literal değerlerle
kubectl create configmap app-config \
  --from-literal=APP_ENV=production \
  --from-literal=LOG_LEVEL=info

# Dosyadan
kubectl create configmap nginx-config \
  --from-file=nginx.conf=./nginx.conf

# Tüm dosyaları bir dizinden
kubectl create configmap app-configs \
  --from-file=./configs/
```

```bash
# ConfigMap'i görüntüle
kubectl get configmap app-config
kubectl describe configmap app-config
kubectl get configmap app-config -o yaml
```

### ConfigMap'i Pod'a Bağlama

**Yöntem 1: Environment Variable olarak**

```yaml
spec:
  containers:
  - name: app
    image: my-app:1.0
    env:
    # Tek bir key al
    - name: APP_ENVIRONMENT
      valueFrom:
        configMapKeyRef:
          name: app-config    # ConfigMap adı
          key: APP_ENV        # Key adı
    - name: LOG_LEVEL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: LOG_LEVEL
```

**Yöntem 2: Tüm ConfigMap'i env olarak yükle**

```yaml
spec:
  containers:
  - name: app
    image: my-app:1.0
    envFrom:
    - configMapRef:
        name: app-config    # Tüm key'ler env var olur
```

**Yöntem 3: Volume olarak mount et (Dosya içeriği için)**

```yaml
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    volumeMounts:
    - name: nginx-config-vol
      mountPath: /etc/nginx/conf.d   # Dosyalar buraya yazılır
      readOnly: true

  volumes:
  - name: nginx-config-vol
    configMap:
      name: nginx-config              # ConfigMap adı
```

```bash
# Container içinden doğrula
kubectl exec -it <pod-name> -- cat /etc/nginx/conf.d/nginx.conf
```

> 💡 **Volume Mount Avantajı:** ConfigMap güncellendiğinde volume olarak mount edilen dosyalar otomatik güncellenir (birkaç dakika içinde). Environment variable olarak bağlananlar için pod yeniden başlatmak gerekir.

## Secret

Şifreler, API anahtarları, TLS sertifikaları gibi **hassas verileri** saklar. ConfigMap'ten farkı:

- Veriler **base64 ile encode** edilir (şifrelenmiş değil, sadece kodlanmış!)
- etcd'de ayrı, kısıtlı erişimle saklanabilir
- RBAC ile erişim kısıtlanabilir
- Bazı sistemlerde encryption at rest yapılandırılabilir

> ⚠️ **Önemli:** base64 encoding şifreleme DEĞİLDİR. `echo "mypassword" | base64` herkes çözebilir. Secret güvenliği için RBAC, etcd encryption, ve harici secret yönetim sistemleri (Vault, External Secrets Operator) kullanılmalıdır.

### Secret Tipleri

| Tip | Kullanım |
|-----|---------|
| `Opaque` | Genel amaçlı (varsayılan) |
| `kubernetes.io/tls` | TLS sertifikası ve private key |
| `kubernetes.io/dockerconfigjson` | Docker registry kimlik bilgileri |
| `kubernetes.io/service-account-token` | Service account token'ları |

### Secret Oluşturma

**Yöntem 1: YAML ile (base64 encode gerekli)**

```bash
# Önce değerleri encode et
echo -n "my-db-password" | base64
# bXktZGItcGFzc3dvcmQ=

echo -n "admin" | base64
# YWRtaW4=
```

```yaml
# db-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  username: YWRtaW4=           # admin (base64)
  password: bXktZGItcGFzc3dvcmQ=  # my-db-password (base64)
```

**Yöntem 2: stringData (otomatik encode)**

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:              # base64 encode etmeden yaz!
  username: admin
  password: my-db-password
```

**Yöntem 3: Komut satırından**

```bash
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=my-db-password

# TLS Secret
kubectl create secret tls my-tls-secret \
  --cert=./tls.crt \
  --key=./tls.key

# Docker registry secret
kubectl create secret docker-registry my-registry \
  --docker-server=ghcr.io \
  --docker-username=myuser \
  --docker-password=mytoken
```

```bash
# Secret'i görüntüle
kubectl get secret db-credentials
kubectl describe secret db-credentials  # Değerler gösterilmez!
kubectl get secret db-credentials -o yaml

# Değeri decode et
kubectl get secret db-credentials \
  -o jsonpath='{.data.password}' | base64 -d
```

### Secret'i Pod'a Bağlama

**Yöntem 1: Environment Variable**

```yaml
spec:
  containers:
  - name: app
    image: my-app:1.0
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: password
```

**Yöntem 2: Volume olarak mount**

```yaml
spec:
  containers:
  - name: app
    image: my-app:1.0
    volumeMounts:
    - name: db-creds
      mountPath: /etc/secrets
      readOnly: true            # Güvenlik için readonly!

  volumes:
  - name: db-creds
    secret:
      secretName: db-credentials
      defaultMode: 0400         # Yalnızca owner okuyabilir
```

```bash
# Container içinde dosyalar:
# /etc/secrets/username → admin
# /etc/secrets/password → my-db-password
```

## Gerçek Dünya Örneği: Web Uygulaması

```yaml
# webapp-full.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: webapp-config
data:
  APP_PORT: "8080"
  LOG_LEVEL: "info"
  DATABASE_HOST: "postgres-service"
  DATABASE_PORT: "5432"
  DATABASE_NAME: "myapp"
---
apiVersion: v1
kind: Secret
metadata:
  name: webapp-secrets
type: Opaque
stringData:
  DB_USERNAME: "appuser"
  DB_PASSWORD: "s3cur3P@ss!"
  JWT_SECRET: "my-super-secret-jwt-key-min-32-chars"
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
      - name: webapp
        image: my-webapp:1.0
        ports:
        - containerPort: 8080
        envFrom:                    # ConfigMap'ten tüm env var'lar
        - configMapRef:
            name: webapp-config
        env:                        # Secret'tan belirli değerler
        - name: DB_USERNAME
          valueFrom:
            secretKeyRef:
              name: webapp-secrets
              key: DB_USERNAME
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: webapp-secrets
              key: DB_PASSWORD
        - name: JWT_SECRET
          valueFrom:
            secretKeyRef:
              name: webapp-secrets
              key: JWT_SECRET
```

## ConfigMap vs Secret Karşılaştırması

| Özellik | ConfigMap | Secret |
|---------|-----------|--------|
| Veri türü | Hassas olmayan konfigürasyon | Şifreler, token'lar, sertifikalar |
| Saklama | Düz metin | base64 encoded |
| etcd'de şifreleme | Hayır (varsayılan) | Opsiyonel (encryption at rest) |
| Boyut limiti | 1 MiB | 1 MiB |
| Kullanım | DB host, port, feature flag'ler | API key, şifre, TLS cert |

## 🛠️ Hands-on: Konfigürasyon Yönetimi

**Adım 1:** ConfigMap ve Secret oluşturun:

```bash
# ConfigMap
kubectl create configmap app-config \
  --from-literal=APP_ENV=development \
  --from-literal=LOG_LEVEL=debug \
  --from-literal=DB_HOST=postgres-service

# Secret
kubectl create secret generic app-secrets \
  --from-literal=DB_PASSWORD=localdev123 \
  --from-literal=API_KEY=test-api-key-abc123
```

**Adım 2:** Değerleri doğrulayın:

```bash
kubectl get configmap app-config -o yaml
kubectl get secret app-secrets -o yaml

# Secret'ı decode et
kubectl get secret app-secrets \
  -o jsonpath='{.data.DB_PASSWORD}' | base64 -d
echo  # Newline ekle
```

**Adım 3:** Pod ile test edin:

```bash
cat > test-env-pod.yaml << 'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: env-test
spec:
  restartPolicy: Never
  containers:
  - name: env-printer
    image: busybox
    command: ["env"]
    envFrom:
    - configMapRef:
        name: app-config
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: app-secrets
          key: DB_PASSWORD
EOF

kubectl apply -f test-env-pod.yaml
kubectl logs env-test | grep -E "APP_ENV|LOG_LEVEL|DB_HOST|DB_PASSWORD"
```

**Temizlik:**
```bash
kubectl delete pod env-test
kubectl delete configmap app-config
kubectl delete secret app-secrets
```

## Özet

Bu bölümde konfigürasyon yönetimini öğrendik:

- **ConfigMap:** Hassas olmayan konfigürasyon verileri için (URL'ler, ayarlar, konfigürasyon dosyaları)
- **Secret:** Hassas veriler için (şifreler, API anahtarları, TLS sertifikaları)
- Bağlama yöntemleri: `env`, `envFrom`, `volume mount`
- base64 encoding şifreleme değildir, ek güvenlik önlemleri gerekir
- ConfigMap/Secret değişince volume'lar otomatik güncellenir, env var'lar için pod restart gerekir

Bir sonraki bölümde container'ların verilerini kalıcı hale getirmek için kullandığımız **PersistentVolume ve PVC** kavramlarını öğreneceğiz.
